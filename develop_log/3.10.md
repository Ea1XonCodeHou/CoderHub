# 3.10 开发日志 - AI 对话体验升级方案

## 任务背景

当前 AI 助手（`/ai/assistant`）存在两个核心问题：
1. **UI 层**：没有区分"普通对话"和"深度检索（RAG）"模式的入口，用户体验割裂
2. **RAG 质量**：存储的是文章整体 Embedding，缺少 Chunking、HyDE、混合召回，召回精度低

---

## 一、前端胶囊按钮方案

### 当前状态
- `/ai/chat/stream` — 普通对话（工具调用 Agent）
- `/ai/chat/rag` — RAG 增强对话
- 前端没有任何模式切换 UI，两个接口的选择逻辑不清晰

### 目标交互设计

参考 DeepSeek / 豆包，在输入框底部增加两个胶囊 Toggle 按钮：

```
┌──────────────────────────────────────────────┐
│  输入你的问题...                               │
│                                              │
│  [🔍 深度检索]  [🌐 联网搜索]        [发送 ↑] │
└──────────────────────────────────────────────┘
```

- **默认状态**：两个胶囊均未激活 → 走 `/ai/chat/stream`（普通 Agent 对话）
- **点击"深度检索"**：胶囊高亮激活 → 走 `/ai/chat/rag`（RAG 知识库增强）
- **点击"联网搜索"**：本期不实现，胶囊可点击但 toast 提示"功能即将上线"
- **两个胶囊互斥**：选择一个时另一个自动取消

### 前端需改动的内容
1. **`AIAssistant.vue`**：输入区底部增加两个 `<button>` 胶囊，维护 `activeMode` 状态（`'normal' | 'rag' | 'web'`）
2. **发送逻辑**：根据 `activeMode` 决定调用哪个 API 函数
3. **UI 样式**：激活态胶囊用主色调边框 + 浅背景色，与 DeepSeek 风格一致

## 二、后端 RAG 系统性提升方案

### 当前 RAG 的问题诊断

| 问题 | 现状 | 影响 |
|------|------|------|
| 存储单位 | 整篇文章一个 Embedding | 长文章语义被平均化，召回的是文章而非答案片段 |
| 查询方式 | 直接用原始问题查向量 | 问题与答案的语义空间天然不对称，相关度低 |
| 召回方式 | 仅 Vector Top-K | 无法覆盖精确关键词匹配场景 |

---

### 方案一：Chunking + 10% Overlap（基础必做）

**核心思想：** 把文章拆成多个重叠段落，每段独立 Embedding，ChromaDB 存的是段落而非文章。

**分段策略（贴合技术博客场景）：**

```
文章文本 → 按段落分割 → 合并为 Chunk（约 512 字） → 10% Overlap 滑窗
```

具体规则：
- **目标 Chunk 大小**：400~600 中文字符（约 150~200 tokens）
- **Overlap**：相邻 Chunk 共享约 50 字符（10% of 512）
- **优先按段落边界切割**：不打断一个完整段落，超长段落再按句子切
- **Chunk 元数据**：存 `{articleId, chunkIndex, title, chunkText前50字}`，便于溯源

**ChromaDB 存储结构变化：**
```
Before：1篇文章 = 1条向量记录
After ：1篇文章 = N条向量记录（N = 文章段落数）
```

**改动范围：** `ArticleEmbeddingService.java`（新增 `chunkText()` 方法，修改 `embedArticle()` 逻辑）

---

### 方案二：HyDE（Hypothetical Document Embeddings）

**核心思想：** 用户的问题（短、问句）与文档（长、陈述句）语义空间不对齐，先让 LLM 根据问题生成一段"假设性答案文档"，再用该文档的 Embedding 去检索，效果大幅提升。

**流程对比：**

```
传统流程：
用户问题 → Embedding → 向量检索 → 相关段落 → LLM 回答

HyDE 流程：
用户问题 → LLM生成假设答案（~200字） → Embedding → 向量检索 → 相关段落 → LLM 最终回答
```

**HyDE Prompt（写死在代码里即可）：**
```
请根据以下问题，生成一段约150字的假设性技术文档片段，
不需要完全正确，只需要语义上贴近可能的答案：
问题：{userQuestion}
```

**成本评估：**
- 多一次 LLM 调用（qwen-turbo，速度快、便宜）
- 检索精度提升显著，尤其是技术问题场景

**改动范围：** `AIService.java` 的 RAG 方法入口处，加 HyDE 预处理步骤

---

### 方案三：Top-K Vector + BM25 混合召回

**核心思想：** 向量检索擅长语义相似，BM25 擅长精确关键词。两路并行召回后合并、去重、重排，覆盖更多召回场景。

**本项目的落地方案（务实版）：**

由于文章内容存在 OSS（不在 MySQL），无法直接对正文做全文索引。推荐以下**两路召回策略**：

| 召回路 | 数据源 | 实现方式 | 擅长场景 |
|--------|--------|----------|----------|
| **路1：Vector 语义召回** | ChromaDB | 向量相似度 Top-K=8 | "解释XX概念"、"XX和XX区别" |
| **路2：MySQL 关键词召回** | MySQL `article` 表 | `MATCH(title, summary) AGAINST(keyword IN BOOLEAN MODE)` | "Redis持久化教程"、精确标题词 |

**合并策略（RRF — Reciprocal Rank Fusion）：**
```
每路召回结果各自排名，最终得分 = 1/(rank1+60) + 1/(rank2+60)
按最终得分重排，取 Top-5
```

RRF 是工业界混合检索的主流方案，实现简单，无需额外训练。

**MySQL 全文索引前提：** 在 `article` 表的 `title` + `summary` 字段上建 FULLTEXT 索引（已存在则直接用）：
```sql
ALTER TABLE article ADD FULLTEXT INDEX ft_article_search(title, summary);
```

**改动范围：** `AIService.java` RAG 方法中，新增 MySQL 关键词召回路，加 RRF 合并逻辑

## 三、整体改造后的 RAG 流程

```
用户问题
    │
    ├─[普通对话]──────────────────────► Agent 工具调用流式对话
    │
    └─[深度检索]
         │
         ▼
    1. HyDE: LLM 生成假设答案文档
         │
         ▼
    2. 两路并行召回
         ├── Vector: Embedding(假设文档) → ChromaDB Top-8 Chunks
         └── BM25: 提取关键词 → MySQL FULLTEXT Top-5 Articles
         │
         ▼
    3. RRF 合并 + 去重 → Top-5 最相关段落
         │
         ▼
    4. 注入 System Prompt → LLM 生成最终答案（流式输出）
```

## 五、Chunking 后的数据迁移策略

现有 ChromaDB 中存的是整篇文章 Embedding，改为 Chunk 后需要重新入库：

1. **清空 ChromaDB** 中的 `coderhub_articles` collection
2. **重建 Redis Set** `embedded_articles`（已嵌入文章记录）
3. **后台批量重嵌**：调用管理员接口 `/admin/embedding/batch`（已有，复用即可）

数据迁移期间 RAG 功能暂时降级（返回"知识库更新中"），或保留旧 collection 切换。
